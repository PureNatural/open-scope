# open-scope

## 背景介绍
目前，**OpenDigger** 尚未系统化地存储和管理 GitHub 仓库的 Topic 信息，尤其对于大量缺乏原生标签（Topic）的仓库，存在显著的信息缺失问题。现有的标签体系仅覆盖了部分主流开源项目，涉及云原生、大数据、大前端、数据库等领域，且主要依赖人工标注完成，存在覆盖范围有限、更新滞后等问题。随着开源生态的快速发展，尤其是 AI 相关项目的激增，人工方式已难以满足对全领域开源项目进行深度洞察与系统化分析的需求。为此，迫切需要构建一种高效、智能的技术标签生成与管理机制，以提升开源数据的可检索性、组织性与分析价值。

## 研究内容
针对上述问题，本文拟基于当前主流的大语言模型（LLM），如 **GPT-4** 等，构建自动化的 GitHub Topic 标注与聚类框架。该框架将从仓库的多模态信息（如名称、描述、README 等）中推理并生成符合语义的高质量标签，以零样本方式提升标签的准确性与覆盖度。在此基础上，进一步结合聚类与语义关联分析方法，自动构建更全面、层次化的技术标签体系，从而实现开源项目标签体系的智能扩展与结构化优化。该研究将为 **OpenDigger** 构建完整的技术标签体系提供关键支撑，并为开源生态的全景理解与演化分析奠定数据基础。

本框架采用模块化架构设计，通过四阶段流水线实现基于大语言模型的GitHub仓库主题标注任务。框架工作流程遵循数据驱动范式，各阶段按序衔接形成完整处理闭环，具体架构如图1所示。

框架首先从GitHub平台收集项目数据，其中在数据源的选择过程中采用了基于OpenRank指标的科学筛选方法。OpenRank是一种专门用于评估开源项目活跃度和影响力的量化指标，能够有效反映项目在开源生态中的实际价值和重要性[11]。本研究利用OpenRank指标对2024年度GitHub平台上的全部仓库进行了活跃程度排名计算，并从中选取排名前3000的高质量项目作为研究样本。框架从GitHub平台收集的项目数据包括仓库简介、README文档内容和已有的主题标签等基本信息。考虑到大语言模型的上下文窗口限制，对README文档进行动态截断处理，不保留超过窗口字符数目限制的内容，在确保关键信息不丢失的同时避免超出模型处理能力。

## 评价指标
为全面评估大语言模型在GitHub项目主题标注任务中的性能表现，主要采用了精确率、召回率、F1分数和成功率四个核心指标

## Baseline

实验选择了当前主流的五个大语言模型进行对比评估，包括GPT-4o（后文简称为GPT）、DeepSeek-v3（DeepSeek）、Claude-3-7-Sonnet（Claude）、Gemini-2.0-Flash（Gemini）和Qwen-Plus-Latest（Qwen）。GPT作为OpenAI的旗舰模型具有优秀的通用能力；DeepSeek在代码理解方面表现突出；Claude在文本分析任务中具有良好表现；Gemini具有优秀的推理效率和多模态能力；Qwen作为中文优化模型在多语言场景下具有优势。
